Why we need a new leaderboard now

The vector database market has moved past a simple "who has the highest QPS?" phase. Most production teams now choose managed vector database products under budget, latency, freshness, and operational constraints. In that world, cost often dominates raw throughput: a faster system is not the better choice if it needs meaningfully more monthly spend to satisfy the same workload.

VDBBench Leaderboard v1 still matters. It gives a clean way to compare raw engine behavior across open-source and cloud systems. But it does not fully answer the questions teams ask when they are about to operate a cloud vector database in production. When does newly inserted data become searchable? When is it fully indexed? What happens when a query returns metadata or vectors, not just IDs? What happens when the workload is split across many tenants? And what does each QPS tier actually cost?

CloudLeadboard v2 is designed around those questions. It keeps performance in view, but puts it next to the production behaviors and cost model that determine what a customer can actually deploy.

What CloudLeadboard v2 changes

CloudLeadboard v2 is not a replacement for the v1 raw-performance board. It is a new decision layer for production cloud evaluation. The goal is to make the leaderboard useful for a buyer who is deciding between real managed products, not only for a reader comparing peak benchmark numbers.

Dimension CloudLeadboard v1 CloudLeadboard v2
Main question How fast can this vector database go under benchmark conditions? Which managed cloud product best fits a production workload and budget?
Primary lens Raw search performance: QPS, latency, and recall. Production behavior: readiness, payload sensitivity, tenancy, cold latency, and cost.
Product scope Broad comparison across open-source and cloud vector databases. Focused comparison of production-ready managed cloud offerings.
Insert story Mostly treats loaded data as ready for search benchmarking. Measures the path from insert completion to fully searchable and fully indexed.
Search response Primarily evaluates search result quality and speed. Separates IDs-only responses from metadata and raw-vector payload responses.
Tenant model Usually reads as a single-dataset, single-namespace benchmark. Adds multi-tenant workload shape with Cohere 10M split across 1,000 tenants.
Cold behavior Warm benchmark loops can hide first-query latency. Reports cold latency, warm latency, and cold/warm ratio directly.
Cost model Cost is secondary to performance ranking. Cost is a first-class result through QPS curves, Pareto points, and crossovers.
How to read it Use it to understand engine capability and raw performance envelopes. Use it to choose the product and configuration that fits a real traffic tier.

This matters because a single top-line QPS ranking can hide important tradeoffs. A product can look strong on peak throughput while losing ground on recall, tail latency, cold-start behavior, or cost at sustained production QPS. Conversely, a lower-cost profile may be exactly right when the workload can tolerate its storage or cold-latency characteristics.

The new tests we added

Version 2 introduces cloud-oriented cases in VectorDBBench. Each case is designed to catch a production behavior that a plain QPS leaderboard can miss. The excerpts below are intentionally selective: they show the shape of the finding, while the full result repository remains the source for complete matrices and reproduction details.

CloudInsertCase: when writes are actually ready

CloudInsertCase separates client-side insert completion from the states users actually care about: searchable and indexed. That makes insert a two-part question: how fast can a service accept the rows, and how long until those rows are safe to search with indexed performance?

Batch size changes two different numbers

On LAION 100M, larger batches show turbopuffer's best ingest strength clearly. With backpressure disabled, it reaches 14.3K rows/s at batch 5k and 15.8K rows/s at batch 10k, ahead of the Zilliz rows on raw acceptance rate. That is useful for teams optimizing a one-way upload path.

The ready-time view is the more production-oriented signal. Zilliz Cloud Capacity 12CU finishes the searchable-and-indexed path in about 3.1-3.2 hours across the tested batch sizes, while Zilliz Tiered 4CU stays around 4.2-4.3 hours. turbopuffer improves sharply as batch size grows, but its fastest ingest mode still lands at 8.16 hours in the batch-10k run. Pinecone Serverless is included now as a baseline from the ready data; in these insert runs it trails both on throughput and total ready time.

Cost changes the insert decision

Ingestion is also a cost event. A system that accepts rows quickly can still be expensive for a backfill, migration, or recurring refresh. The cost excerpt makes the Zilliz tradeoff easier to see: the full-load write cost is modeled as CU time, while turbopuffer and Pinecone include usage-metered write charges.

CloudPayloadSearchCase: search with real response bodies

Production search often returns more than IDs. CloudPayloadSearchCase measures the cost of returning scalar metadata and raw vectors under unfiltered and filtered search. turbopuffer deserves credit here: in the unfiltered vector-payload excerpt, unpinned turbopuffer reaches 382 QPS, ahead of Zilliz Capacity 12CU at 229 QPS.

The Zilliz advantage shows up when throughput is read together with recall and tail behavior. Zilliz Capacity keeps stronger recall in the unfiltered vector run, and it is much cleaner on highly selective integer filters: at the 99.9% integer-filter/vector point, Zilliz Tiered and Capacity are near the 900-950 QPS range while turbopuffer is around 45 QPS with multi-second p99. The release-note takeaway is not that one system wins every row; it is that payload and filter shape can completely change the ranking.

CloudMultitenantSearchCase: SaaS-shaped traffic

CloudMultitenantSearchCase splits Cohere 10M across 1,000 tenants and sends queries to random tenants. This is where turbopuffer has its clearest raw-performance edge in the current data. Across the matched multitenant matrix, it leads the max-QPS rows, with IDs-only and scalar-label payloads commonly landing around the 3.9K-4.6K QPS band.

Zilliz Cloud Capacity 2CU is still the stronger Zilliz profile and stays much more usable than Tiered for vector payloads. Capacity reaches 371 QPS on unfiltered vector returns and 1,306 QPS on the highly selective integer-filter/vector excerpt, while the refreshed Tiered 1CU vector rows are materially stronger on selective filters but still trail on heavier vector-return cases. This case therefore gives turbopuffer a fair win on namespace-shaped throughput, while also showing why a fixed-capacity Zilliz deployment remains relevant once cost, recall, and operational guarantees are layered in.

CloudColdLatencyCase: the first query after idle

Cold latency is easy to hide in warm benchmark loops, but users feel it immediately. In the LAION 100M ids-only cold-latency run, Zilliz Capacity 12CU has the cleanest accepted cold-tail result: 22.8 ms cold p99 and 11.5 ms cold average. Unpinned turbopuffer shows a much larger cold-start profile, including a 2.05s first cold query.

turbopuffer's pinned two-replica mode is worth calling out, because it materially improves the result: 58.3 ms cold p99 and a 64.2 ms first cold query. That is a meaningful strength for customers willing to operate a pinned mode. Zilliz Capacity still has the lower cold tail in this excerpt, while Zilliz Tiered is marked for rebench because its cold tail is much higher than warm.

The cost view that actually helps buyers

The Pareto cost board is the most important new decision surface in CloudLeadboard v2. It shows the estimated cost of each product at different target QPS levels and marks the products that are Pareto-optimal under the benchmark's latency, recall, payload, and readiness constraints.

In the current LAION 100M compute model, turbopuffer is compelling for very low sustained QPS. That is a legitimate strength of pay-per-query serverless pricing. But once traffic crosses the low single-digit QPS range, Zilliz Cloud's flat CU-hour economics become the stronger cost story. The current crossover is at 8 QPS: from that point onward, the cheapest reachable Zilliz option is less expensive than turbopuffer in this model.

This is exactly why v2 needs a cost board. A customer should not have to infer the buying decision from a raw QPS table. The board should say where turbopuffer's low-traffic advantage applies, where Zilliz Tiered becomes the more economical active-production option, and where Zilliz Capacity is the better fit for higher-QPS or latency-sensitive serving.

What to keep in mind

This is a release note, not the full benchmark report. The job here is to introduce the new leaderboard, explain why the cases matter, and make the direction of the current findings clear. Detailed tables, reproduction commands, raw JSON paths, and edge-case analysis belong in the benchmark report and source artifact repository.

  • Pricing changes over time, so every cost chart needs the pricing date and region.
  • Vendor configuration choices can change results, especially for managed products with scaling controls or serverless behavior.
  • Fully indexed status may be exposed differently by product; the report must document how it is detected or inferred.
  • The current multi-tenant workload uses Cohere 10M split across 1,000 tenants; it is useful, but it does not represent every tenant distribution.
  • Cold latency depends on cache state, idle windows, and storage architecture, so the test definition must remain strict.
  • Cost Pareto winners are workload-specific and should always be read together with latency, recall, and payload constraints.
