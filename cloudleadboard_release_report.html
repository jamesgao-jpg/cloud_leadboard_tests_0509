Why we need a new leaderboard now

The vector database market has moved past a simple "who has the highest QPS?" phase. Most production teams now choose managed vector database products under budget, latency, freshness, and operational constraints. In that world, cost often dominates raw throughput: a faster system is not the better choice if it needs meaningfully more monthly spend to satisfy the same workload.

VDBBench Leaderboard v1 still matters. It gives a clean way to compare raw engine behavior across open-source and cloud systems. But it does not fully answer the questions teams ask when they are about to operate a cloud vector database in production. When does newly inserted data become searchable? When is it fully indexed? What happens when a query returns metadata or vectors, not just IDs? What happens when the workload is split across many tenants? And what does each QPS tier actually cost?

CloudLeadboard v2 is designed around those questions. It keeps performance in view, but puts it next to the production behaviors and cost model that determine what a customer can actually deploy.

What CloudLeadboard v2 changes

CloudLeadboard v2 is not a replacement for the v1 raw-performance board. It is a new decision layer for production cloud evaluation. The goal is to make the leaderboard useful for a buyer who is deciding between real managed products, not only for a reader comparing peak benchmark numbers.

Dimension CloudLeadboard v1 CloudLeadboard v2
Main question How fast can this vector database go under benchmark conditions? Which managed cloud product best fits a production workload and budget?
Primary lens Raw search performance: QPS, latency, and recall. Production behavior: readiness, payload sensitivity, tenancy, cold latency, and cost.
Product scope Broad comparison across open-source and cloud vector databases. Focused comparison of production-ready managed cloud offerings.
Insert story Mostly treats loaded data as ready for search benchmarking. Measures the path from insert completion to fully searchable and fully indexed.
Search response Primarily evaluates search result quality and speed. Separates IDs-only responses from metadata and raw-vector payload responses.
Tenant model Usually reads as a single-dataset, single-namespace benchmark. Adds multi-tenant workload shape with Cohere 10M split across 1,000 tenants.
Cold behavior Warm benchmark loops can hide first-query latency. Reports cold latency, warm latency, and cold/warm ratio directly.
Cost model Cost is secondary to performance ranking. Cost is a first-class result through QPS curves, Pareto points, and crossovers.
How to read it Use it to understand engine capability and raw performance envelopes. Use it to choose the product and configuration that fits a real traffic tier.

This matters because a single top-line QPS ranking can hide important tradeoffs. A product can look strong on peak throughput while losing ground on recall, tail latency, cold-start behavior, or cost at sustained production QPS. Conversely, a lower-cost profile may be exactly right when the workload can tolerate its storage or cold-latency characteristics.

The new tests we added

Version 2 introduces cloud-oriented cases in VectorDBBench. Each case is designed to catch a production behavior that a plain QPS leaderboard can miss. The excerpts below are intentionally selective: they show the shape of the finding, while the full result repository remains the source for complete matrices and reproduction details.

CloudInsertCase: when writes are actually ready

CloudInsertCase separates client-side insert completion from the states users actually care about: searchable and indexed. That makes insert a two-part question: how fast can a service accept the rows, and how long until those rows are safe to search with indexed performance?

Batch size changes two different numbers

Larger batches show turbopuffer's raw ingest strength: with backpressure disabled, it reaches 14.3K rows/s at batch 5k and 15.8K rows/s at batch 10k.

Ready time changes the readout. Zilliz Capacity 12CU reaches searchable-and-indexed in about 3.1-3.2 hours, Tiered 4CU in 4.2-4.3 hours, while turbopuffer's fastest batch-10k path lands at 8.16 hours.

Pinecone is the slower bulk-intake baseline in this excerpt, with ready time around 72-112 hours. Once accepted, though, its extra searchable/indexed wait is effectively zero in these runs.

The product read is simple: Zilliz fits fresh-data workflows, turbopuffer fits large accepted backfills with a longer ready window, and Pinecone fits lower-volume serverless ingestion.

Cost changes the insert decision

Ingestion is also a cost event. In the LAION 100M model, dedicated Zilliz capacity keeps the one-time full-load cost around $18-21 because the write work is spread over CU-hours.

Serverless write models meter usage directly. Here, turbopuffer batch-10k bp-off is modeled at $308, while Pinecone is about $1.18K. Serverless lowers operational friction, but not every large write path is cheaper.

For buyers, the split is recurring production refresh versus occasional or spiky use. Frequent RAG, catalog, or agent-memory refreshes benefit from dedicated capacity; lighter workloads may still prefer serverless operations.

CloudPayloadSearchCase & MultitenantSearchCase: search after data is ready

What this case measures

The leaderboard now treats search as one combined decision surface. Payload search tests single-tenant LAION 100M with IDs, metadata, or vector payloads; multitenant search tests Cohere 10M split across 1,000 tenants.

Design logic

Both modes use the same readout: max concurrent QPS, p99 latency, payload shape, and recall where available. This keeps the comparison consistent when response size, filters, or tenant routing change the workload.

What we found

In single-tenant LAION 100M, turbopuffer leads broad unfiltered retrieval at about 395-400 QPS and still reaches 382.2 QPS with vector payloads. Zilliz Capacity 12CU is close on IDs/scalars and has stronger unfiltered recall at 0.9723.

Selective filters flip the story. At the 99.9% integer-filter/vector point, Zilliz Tiered 4CU reaches 955.7 QPS and Capacity 12CU reaches 933.0 QPS; turbopuffer unpinned drops to 45.1 QPS with about 7s p99.

In multitenant mode, turbopuffer is strongest: 3,855 QPS on unfiltered IDs-only and 3,835 QPS on the selective integer-filter/vector row. Zilliz Capacity 2CU remains the stronger Zilliz profile; Pinecone is a detached low-ops reference rather than a normalized high-concurrency comparison.

The product read is workload-shaped: turbopuffer fits broad retrieval and many lightweight tenants; Zilliz fits filtered, permissioned, heavier-tenant, or recall-sensitive deployments. Pinecone remains the low-ops serverless baseline.

CloudColdLatencyCase: the first query after idle

What this case measures

This case isolates first-query latency after idle. It uses LAION 100M, topK 100, and ids-only responses, then compares a cold serial pass with the same query shape after warmup.

What we found

Zilliz Capacity 12CU has the cleanest accepted cold profile: first request and tail latency stay close to the warm path, which is the cleanest story for interactive AI features.

turbopuffer needs two reads. Unpinned has a visible cold miss; pinned two-replica mode removes most of that penalty, but with a different cost and capacity posture.

Pinecone is useful as a low-ops baseline: cold and warm tails are similar, but both are around one second. Zilliz Tiered 4CU stays in the chart for transparency and should be rebenched.

The cost view that actually helps buyers

Cost is modeled against achievable QPS, not shown as a detached price sheet

What this view measures

The Pareto board turns benchmark results into a buying surface. It estimates cost at target QPS levels and marks products that are Pareto-optimal under measured performance constraints.

Design logic

The board combines measured QPS boundaries with product price models. For LAION 100M query-only, each line stops where the leaderboard measured or capped that product's serving range, then applies Zilliz CU-hour pricing, turbopuffer query/storage or pinned pricing, and Pinecone read-unit/storage pricing.

This makes the chart a simulation over real benchmark boundaries rather than a detached price calculator. The best-choice line is the lower-cost envelope across reachable options.

What we found

In the LAION 100M query-only model, turbopuffer wins at very low sustained QPS. The crossover is 8 QPS; from that point onward, Zilliz Tiered 4CU's flat CU-hour cost is cheaper than turbopuffer's usage-metered query line.

Zilliz gets more competitive as QPS rises because serving cost is mostly flat once provisioned. Tiered is the economical active-production line after crossover; Capacity matters when buyers need a stronger serving envelope, better cold behavior, or a more predictable enterprise posture.

Pinecone remains useful as a low-ops serverless reference, but in this LAION 100M board its read-unit pricing keeps it from becoming the cheapest line. The takeaway is conditional: serverless can win when QPS stays low; provisioned Zilliz wins once utilization is high enough.

Current board state

This is why v2 needs a cost board: it shows where serverless low-traffic economics apply, where Zilliz Tiered becomes the economical active-production option, and where Capacity fits higher-QPS or governance-heavy serving.

What to keep in mind

CloudLeadboard v2 is meant to make cloud vector database evaluation feel closer to the way teams actually buy and operate these systems. Peak QPS still matters, but it is no longer enough on its own. The useful question is whether a product can meet the workload's latency, freshness, payload, tenancy, recall, and cost requirements at the same time.

The current results should be read with that context. Pricing can change by date and region, and managed-product configuration choices can shift the outcome, especially for products with scaling controls, serverless behavior, or pinned capacity modes. Readiness and fully indexed status also need careful documentation because each product exposes or implies those states differently.

The workloads are intentionally specific, not universal. Cohere 10M across 1,000 tenants is a useful SaaS-shaped test, but it does not represent every tenant distribution. Cold latency depends on cache state, idle windows, and storage architecture, so the definition has to stay strict. And every cost Pareto result is workload-specific: it should always be read together with latency, recall, payload shape, and measured serving limits.

That is the main reason this release matters. CloudLeadboard v2 gives us a way to compare managed vector databases as deployable products, not just benchmark entries. The next step is to keep expanding the case matrix and make the full benchmark report carry the reproduction details behind each chart.
